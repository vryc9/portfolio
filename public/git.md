# Définition
Git est un système de gestion de versions décentralisé (VCS) permettant de suivre l'évolution du code source, de collaborer à plusieurs et de revenir en arrière en cas d'erreur.

# Réalisations
Tout au long de ma formation et de mon alternance, j’ai utilisé la technologie Git sur divers projets mais c’est durant mon alternance que j’ai pu utiliser Git avec une utilisation plus poussée ce qui m’a permis de gagner en compétence. J’ai principalement utilisé Git avec l’outil Gitlab qui permet de stocker des projets sur un dépôt distant. 

Dans le cadre d'un projet en entreprise, des développeurs désignés comme « Référents BDD » étaient responsables de l'exécution des scripts PostgreSQL et Oracle sur nos bases de données. Leur processus reposait sur un script JavaScript qui récupérait les derniers scripts ajoutés à la branche principale, puis extrayait les URL GitLab associées avant de les soumettre manuellement à Jenkins pour exécution. Cette procédure était chronophage et source d’erreurs. Pour résoudre ce problème, j'ai configuré des webhooks GitLab sur les projets concernés en ciblant les branches spécifiques et en sélectionnant l'événement « Push Events ». Ainsi, à chaque push d'un développeur, l’ API reçoit automatiquement les informations des commits et déclenche le traitement associé. Grâce à cette configuration, l'intégralité du processus est devenue automatique, supprimant les interventions manuelles et réduisant le risque d'oubli.

En entreprise, nous maintenons en parallèle plusieurs versions de nos applications. Il arrive qu'un développement réalisé sur une version doit être reporté sur une autre version. Dans ce contexte, un merge entre les deux branches est à éviter car il risquerait d'embarquer l'ensemble des différences entre les deux versions. J'utilise donc le cherry-pick directement depuis les merge requests GitLab, ce qui me permet de sélectionner précisément les commits à reporter sans impacter le reste de la branche cible, évitant ainsi toute régression ou effet de bord.

Pour le développement du portfolio, j'ai mis en place un pipeline CI/CD via GitHub Actions pour automatiser le déploiement sur GitHub Pages. J'ai créé et configuré un fichier deploy.yml déclenché à chaque push sur la branche principale. Ce pipeline assure le build de l'application ainsi que déploiement. Cette expérience m'a également permis de découvrir l'écosystème GitHub Actions en complément de GitLab, élargissant ainsi ma maîtrise des outils qui utilisent la technologie Git.

# Autocritique et évolutions
Au départ, la compréhension globale du fonctionnement a été complexe en raison du nombre important de commandes disponibles. Ma progression a été constante au fil des années. Aujourd’hui je considère avoir atteint un niveau intermédiaire ce qui me permet d’être autonome pour résoudre la plupart des problématiques rencontrées.

Avec mon expérience, je recommande de commencer avec les bases de Git en apprenant les commandes de base et comprendre ce que ça fait derrière sur un projet en autonomie. Par la suite, apprendre à utiliser Git dans un contexte où plusieurs personnes travaille sur un même projet. Cela va permettre de rencontrer des problèmes de conflit, avoir une version du code à jour, qui je trouve est très formateur. Dans mon profil de développeur, cette compétence est indispensable car aujourd’hui pratiquement tous les projets sont maintenus et versionné par Git. Avoir cette compétence me semble indispensable pour collaborer en équipe.

Pour ma marge de progression, je souhaite approfondir mes connaissances sur GitLab CI/CD ou Github afin de maîtriser l'automatisation complète du déploiement des applications (Pipeline de test, build et déploiement). Pour cela, je m’appuie sur la documentation officielle de Github / GitLab et j’expérimente à travers des projets tel que le portfolio avec la mise en place de Github Pages.
