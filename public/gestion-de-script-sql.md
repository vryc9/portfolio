### Présentation 
Au sein de notre équipe, nous avons des développeurs désignés comme "Référents BDD" qui sont responsables de l'exécution des scripts PostgreSQL et Oracle afin de mettre à jour nos bases de données. Pour ce faire, ils utilisent un **script JavaScript** qui récupère les derniers scripts ajoutés à la branche principale du projet, puis ils extraient les URL GitLab associées à ces scripts. Ensuite, les référents BDD les soumettent à **l’outil Jenkins**, qui se charge de l'exécution des scripts. Mais cette solution prend beaucoup de temps à mettre en place. 

Pour répondre à ce besoin, j'ai développé un écran dans l'application interne à l'équipe qui **récupère automatiquement les scripts SQL poussés sur GitLab via un Webhook** les affiche aux référents BDD avec un système de suivi d'état (à traiter, traité, anomalie), et génère des fichiers d'export directement exploitables dans Jenkins.

### Objectif 
Avant ce projet, les référents BDD **exécutaient manuellement un script JavaScript** pour extraire les URLs des scripts SQL récemment ajoutés, avant de les soumettre à Jenkins. Cette procédure était devenue chronophage à mesure que le nombre de scripts augmentait.

L'objectif était de **livrer une interface dédiée dans le wall, intégrant la récupération automatisée des scripts via Webhook GitLab**, un suivi d'état pour chaque script, et une fonctionnalité d'export. L'enjeu était donc de faire gagner du temps aux référents sur une tâche récurrente, et avoir une traçabilité visuelle de ce qui avait été traité ou non.

Côté entreprise, les principaux risques portaient sur la **fiabilité** : un script oublié par le Webhook et un état mal renseigné pouvait induire un référent en erreur. Côté personnel, il s'agissait de ma première expérience avec les Webhooks et la gestion de flux temps réel, ce qui représentait un risque de **sous-estimation de la complexité**. Un dernier risque, que je n'avais pas identifié à ce stade, portait sur l'adoption réelle de l'outil sur la durée.

### Etapes
Afin d'optimiser leur efficacité, un **nouvel écran** sur le wall ( Application interne à l’équipe ) doit être mis en place, offrant une visualisation des derniers scripts récupérés depuis GitLab. Depuis cette interface, un mécanisme d'exportation génèrera un fichier texte pour chaque type de script, contenant les URLs correspondantes. De plus, un système d'états doit être intégré pour chaque script : "à traiter", "traité", et "anomalie", assurant un suivi visuel.

Par défaut, lorsqu'un script est récupéré, il doit être automatiquement classé comme "à traiter" puisqu'il n'aura pas encore été exécuté sur une base de données. Une fois le script exécuté, le référent met à jour son état. Lors de l'exécution, s’il n’y a pas d’erreur, l'état devient "traité". Cependant, si le script rencontre un problème, le référent le classera en "anomalie" et a la possibilité d'ajouter une description explicative.

Afin de recevoir les événements des commits, Gitlab met à disposition un Webhook. Il envoie une requête à une API, contenant diverses informations telles que le nom de l'auteur du commit, le nom de la branche, les fichiers ajoutés, modifiés ou supprimés. Cette fonctionnalité m’a été utile pour récupérer les fichiers du push et filtrer uniquement les scripts SQL, afin de les afficher sur l'interface. J’ai donc utilisé cette solution qui offre une automatisation complète et autonome de la récupération des derniers scripts, ce qui représente un gain de temps par rapport à l'approche précédente nécessitant l'exécution d'un script JavaScript. 

Avant de commencer le développement, j’ai réalisé une maquette sur Figma, afin d’avoir un premier visuel de l’interface. Pour incorporer chacune des fonctionnalités souhaitées, j’ai effectué des recherches au niveau du design pour avoir l’interface utilisateur la plus optimisée et la plus agréable possible. Pour chaque fonctionnalité, j'ai créé des séquences d'actions correspondant aux différentes actions possibles pour l'utilisateur. Cela permet d’avoir un premier visuel sur les différents rendus sur chaque action. Le retour des utilisateurs était très important c’est donc pour cela que j’ai pu partager le lien Figma menant à la maquette afin d’avoir directement les retours des utilisateurs sur les éléments à changer. 

J’ai ensuite configuré les Webhooks sur les différents projets. J’ai d’abord indiqué l’url vers mon API pour recevoir les différents événements. Par la suite, j’ai configuré la branche sur laquelle je voulais recevoir les informations en indiquant juste le nom de celle-ci. Puis pour finir, j’ai coché la case « Push Events », qui indique au Webhook qu’il faut qu’il envoie les informations lorsqu’une personne push sur la branche indiquée. Avec cette configuration, je récupère de manière automatisée les différents fichiers lors de chaque push effectué sur la branche indiquée.

Pour les fonctionnalités, j’ai mis en place une logique d’extraction qui m’a servi à manipuler les données envoyées par le Webhook pour ensuite les stocker en base de données. Pour le côté utilisateur, j’ai développé un système d’export comme précisé auparavant.  La complexité dans ce projet a été de gérer le temps réel des informations. J’ai donc, utilisé les Observables, mise à disposition par Angular qui m’ont aidé lors du développement. 

### Acteurs et interactions 
Tout au long de ce projet j’ai communiqué avec les référents pour leur montrer mon avancée et prendre en compte leur retour sur les fonctionnalités ou sur les maquettes. Pour la gestion de projet, j’ai également été accompagné par ma responsable d’alternance, avec qui nous estimions les user stories lors de notre point de synchronisation hebdomadaire. 

### Résultats et lendemain
A la fin du développement, l'outil **a été bien accueilli par les référents BDD**. Il supprimait l'étape manuelle d'exécution du script JavaScript et offrait, une visibilité qui n'existait pas auparavant grâce au système d'états et à la gestion des scripts dans une seule interface. Les exports générés étaient exploitables dans Jenkins, ce qui fluidifiait le jeu des scripts sur les bases de données. Pour moi, ce projet a été très formateur sur plusieurs plans. Sur le plan technique, j'ai découvert la configuration et l'exploitation des Webhooks GitLab, ainsi que la gestion des flux asynchrones avec les Observables d'Angular.

Dans l'immédiat suivant la fin du développement, l'outil a été utilisé par les référents et a effectivement réduit le temps consacré à la récupération des scripts. Aujourd'hui, l'outil est de moins en moins utilisé par l'équipe, malgré son bon fonctionnement technique. Nous nous sommes rendu compte qu'une contrainte persistait : les référents devaient manuellement reconfigurer le Webhook sur GitLab pour changer la branche cible lors des changements de contexte. Cette action répétitive a fini par décourager l'usage de l'outil sur le long terme.

Avec le recul, j'aurais dû concevoir l'architecture différemment pour éviter cette configuration manuelle. Par exemple, j'aurais pu configurer le Webhook pour écouter toutes les branches, et ajouter un filtre dynamique directement dans mon interface Angular pour laisser le référent choisir sa branche à la volée.
